# 워크스페이스 결정 기록

## 목적

이 문서는 여러 프로젝트와 공용 문서 체계에 함께 영향을 주는 구조·운영 합의를 기록합니다. 프로젝트 제품 결정은 해당 프로젝트의 PRD나 아키텍처 문서에 남기며, 작은 문구 수정이나 쉽게 되돌릴 수 있는 편집은 여기에 기록하지 않습니다.

## 기록 기준

다음 중 하나에 해당할 때만 항목을 추가합니다.

- 저장소의 디렉터리 구조나 필수 문서를 바꾸는 결정.
- 여러 프로젝트가 따라야 하는 문서 흐름이나 추적성 규칙.
- 검증 스크립트가 허용하거나 금지하는 범위를 바꾸는 결정.
- 특정 도구나 플랫폼에 대한 저장소 차원의 의존 여부.
- 이후 기여자가 이유를 모르면 되돌리기 쉬운 결정.

각 결정은 `Active`, `Superseded`, `Revisit` 중 하나로 관리합니다. 기존 결정을 바꿀 때는 항목을 삭제하지 않고 새 결정에서 이전 ID를 연결합니다.

## 결정 상세

### W-001 자동 탐색형 에이전트 지침을 제공하지 않음

- 결정: 저장소에 `AGENTS.md`, `AGENTS.override.md` 같은 자동 탐색형 지침 파일을 커밋하지 않습니다.
- 이유: 기여자가 이미 사용하는 전역·상위 지침과 예기치 않게 합쳐져 도구 동작을 바꿀 수 있습니다.
- 대안: 저장소 규칙은 `CONTRIBUTING.md`와 공용 문서 표준에서 명시적으로 읽습니다. 개인 지침은 사용자가 로컬에서 관리합니다.
- 영향: 에이전트가 저장소 규칙을 자동으로 읽는다고 가정하지 않습니다.
- 재검토 조건: 팀이 특정 에이전트를 공식 표준 도구로 채택하고 저장소 수준 자동 지침에 합의할 때.

### W-002 clone 위치 독립성

- 결정: 내부 링크는 상대 경로를 사용하고 스크립트는 자신의 위치에서 저장소 루트를 계산합니다.
- 이유: 사용자명, 드라이브, 운영체제, clone 디렉터리가 달라도 같은 저장소가 동작해야 합니다.
- 대안: 문서 설명에만 이식성을 맡기지 않고 검증 스크립트가 절대 사용자 경로와 절대 로컬 링크를 확인합니다.
- 영향: 로컬 환경을 설명해야 할 때는 실제 절대 경로 대신 저장소 루트 기준 예시를 사용합니다.
- 재검토 조건: 외부 시스템이 절대 경로를 요구하는 산출물이 추가될 때 해당 파일 유형만 예외로 검토합니다.

### W-003 제품 프로젝트 식별 방식

- 결정: 제품 프로젝트는 루트 `README.md`의 `project-index` 범위에 등록하고 프로젝트 `README.md`, `docs/README.md`, `docs/prd.md`를 갖습니다.
- 이유: 모든 최상위 디렉터리를 프로젝트로 간주하면 향후 `assets`, `examples`, `tools` 같은 공용 디렉터리를 잘못 검증합니다.
- 대안: `docs/prd.md`가 있는 미등록 디렉터리도 프로젝트 후보로 탐지해 인덱스 누락을 보고합니다.
- 영향: 새 제품 프로젝트를 추가할 때 루트 프로젝트 표와 세 필수 문서를 함께 갱신합니다.
- 재검토 조건: 제품 프로젝트가 아닌 라이브러리·데이터셋 등 다른 일급 산출물 유형을 저장소에서 관리할 때.

### W-004 템플릿의 선택적 적용

- 결정: 템플릿은 출발점이며 모든 섹션과 기본 평가 조건을 강제로 적용하지 않습니다.
- 이유: 웹 화면, 상업 시장, 고정 기간, 고정 데모 시간 같은 기본값이 다양한 프로젝트를 현재 사례에 과적합시킬 수 있습니다.
- 대안: 적용되지 않는 항목은 이유와 함께 `해당 없음`으로 표시하거나 프로젝트의 기존 형식에 맞게 생략합니다.
- 영향: 팀은 평가 전에 목표 직무, 기간, 인력, 전달 방식을 합의하고 템플릿 기본값을 조정합니다. 기존 문서에는 하위 문서가 참조해야 할 때부터 추적 ID를 추가하며 형식만 맞추기 위한 일괄 개편은 하지 않습니다.
- 재검토 조건: 반복 프로젝트에서 반드시 필요한 최소 공통 필드가 충분한 사례로 확인될 때.

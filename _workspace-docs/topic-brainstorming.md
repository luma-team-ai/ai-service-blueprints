# 일반 사용자용 포트폴리오 프로젝트 주제 브레인스토밍

## 목적

이 문서는 일반 사용자가 개발 지식 없이도 사용할 수 있는 새 제품 주제를 발굴하고, 후보 비교부터 선택·보류·배제, 프로젝트 PRD 승격까지 추적합니다. 개별 후보를 구체화할 때는 [`templates/topic-brainstorm.md`](templates/topic-brainstorm.md)를 사용합니다.

기존에 취업 준비 영역(자기소개서·이력서, 직무 매칭, 면접 연습)과 MeetingHub가 회의 업무를 다루므로, 새 주제는 그 영역을 반복하지 않으면서 생활 속에서 사용자가 직접 겪는 문제를 해결해야 합니다.

## 상태

- 해당자: 일반 사용자용 취업 포트폴리오 프로젝트를 만들려는 기존 팀원
- 마지막 업데이트: 2026-06-18
- 상태: Active
- 현재 결정: T-025(review-sentiment) ML 과제 프로젝트 Selected로 승격. 기존 후보 6개는 일반 사용자 제품 트랙 유지.
- 다음 결정: review-sentiment PRD 검토 완료 후 구현 단계 진입. 일반 사용자 후보는 접근 가능한 사용자와 데이터 경로 확인 후 상위 2개로 좁힘.

## 후보 기준

새 후보는 다음 조건을 우선합니다.

1. 개발자나 회사 내부 운영자가 아닌 일반 사용자가 직접 사용합니다.
2. 사용자가 한 문장으로 설명할 수 있는 생활 문제를 해결합니다.
3. 검색 결과를 다시 보여주는 데서 끝나지 않고 등록, 판단, 행동, 확인으로 이어지는 흐름이 있습니다.
4. 실제 사용자 인터뷰나 현실적인 대체 검증이 가능합니다.
5. 개인정보와 안전 위험을 MVP 범위에서 통제할 수 있습니다.
6. 기존 취업 준비 영역과 MeetingHub의 핵심 문제를 반복하지 않습니다.
7. 포트폴리오 기술은 사용자 문제를 해결하는 과정에서 자연스럽게 드러나야 합니다.

## 기존 포트폴리오 기준선

| 프로젝트 | 현재 보여주는 문제 영역 | 새 주제에서 피할 중복 |
| --- | --- | --- |
| 취업 준비 영역 | 취업 준비, 자기소개서·이력서, 직무 매칭, 면접 연습 | 구직 추천, 범용 문서 첨삭, 면접 챗봇 |
| MeetingHub | 회의 기록, 요약·분석, 후속 작업 | 회의 요약, 범용 음성 기록, 협업 회고 생성 |

MeetingHub의 상세 기능과 구현 범위는 이 저장소에 없으므로 최종 선택 전에 실제 프로젝트와 중복을 다시 확인합니다.

## 평가 방식

점수보다 사용자를 먼저 확인합니다. 필요할 때 다음 기준을 1점에서 5점으로 평가하되, 사용자 문제와 검증 경로가 약한 후보를 기술적 화려함만으로 선택하지 않습니다.

| 기준 | 가중치 | 확인 질문 |
| --- | ---: | --- |
| 사용자 문제 명확성 | 25 | 일반 사용자가 반복해서 겪거나 손실을 피하고 싶은 문제인가? |
| 행동 가능한 결과 | 20 | 결과가 사용자의 다음 행동과 직접 연결되는가? |
| 검증 가능성 | 15 | 인터뷰, 관찰, 샘플 데이터로 가장 위험한 가정을 확인할 수 있는가? |
| 데이터 지속 가능성 | 15 | 합법적이고 안정적인 데이터 또는 사용자 입력 경로가 있는가? |

| 포트폴리오 보완 | 10 | 기존 프로젝트와 다른 제품·기술적 증거를 보여주는가? |

## T-019 우리집 제품 리콜·안전 알림

- 상태: Candidate
- 대상 사용자: 가전, 생활용품, 어린이 제품을 구매하고 보관하는 개인 또는 가족.
- 반복 문제: 리콜 공지는 여러 사이트와 뉴스에 흩어져 있고, 사용자는 자신이 가진 제품이 대상인지 뒤늦게 알기 쉽습니다.
- 한 줄 제안: 제품명, 모델명, 사진 또는 인증번호로 보유 제품을 등록하면 공식 리콜 정보와 대조해 대상 여부와 조치 방법을 알려주는 생활 안전 보관함.
- 핵심 흐름: 제품 등록 → 공식 제품·리콜 정보 대조 → 일치 가능성 표시 → 사용 중지·교환·문의 행동 안내 → 처리 상태 기록.
- MVP: 제품 수동 등록, 모델명 검색, 공식 리콜 목록 대조, 알림, 처리 완료 체크.
- 사용자에게 보이는 결과: `대상 가능성`, 일치 근거, 공식 원문, 지금 해야 할 행동.
- 기술적 증거: 공공 API 연동, 검색·정규화, 알림, 변경 감지, 불확실한 매칭 UX.
- 가장 위험한 가정: 사용자가 제품을 등록하는 수고를 감수할 만큼 리콜 알림의 가치를 느끼는가.
- 첫 검증: 최근 리콜 제품 20개로 모델명 매칭 정확도를 확인하고 생활용품 구매자에게 등록 흐름을 테스트합니다.
- 주요 위험: 비슷한 모델명을 확정적으로 일치시키지 않고 공식 원문과 제조사 확인 경로를 함께 제공해야 합니다.
- 데이터 후보: [Safety Korea 제품안전정보센터](https://www.safetykorea.kr/)의 리콜 정보와 Open API.

## T-020 영수증·반품·보증 기한 보관함

- 상태: Candidate
- 대상 사용자: 온라인 쇼핑과 오프라인 구매가 잦고 영수증, 주문 내역, 보증서를 여러 곳에 보관하는 사람.
- 반복 문제: 반품 가능일, 교환 조건, 보증 종료일, 영수증 위치를 잊어 필요한 권리를 사용하지 못합니다.
- 한 줄 제안: 영수증이나 주문 화면을 저장하면 상품, 구매처, 결제일, 반품·보증 기한을 추출해 구매 후 해야 할 일을 정리하는 개인 보관함.
- 핵심 흐름: 사진·PDF·직접 입력 → 구매 정보 확인·수정 → 반품·보증 마감 계산 → 알림 → 반품·수리·보관 완료 처리.
- MVP: 영수증 OCR, 항목 수정, 사용자가 확인한 기한 등록, D-day 목록, 원본 증빙 보관, 내보내기.
- 사용자에게 보이는 결과: 이번 주에 확인할 반품, 보증 만료 예정 제품, 필요한 영수증과 문의 정보.
- 기술적 증거: OCR, 구조화 추출 검증, 파일 보안, 알림, 검색, 사용자 수정 이력.
- 가장 위험한 가정: 자동 추출이 완벽하지 않아도 사용자가 구매 후 관리 시간을 줄였다고 느끼는가.
- 첫 검증: 서로 다른 형태의 실제·합성 영수증 30개로 추출과 수정 시간을 비교합니다.
- 주요 위험: 판매처별 정책을 자동으로 단정하지 않고 사용자가 확인한 조건과 공식 원문을 구분해야 합니다.
- 데이터 경로: 사용자 제공 영수증과 주문 화면을 기본으로 하며 외부 쇼핑몰 계정 연동은 MVP에서 제외합니다.

## T-021 냉장고 소비기한·식품 회수 알림

- 상태: Candidate
- 대상 사용자: 장을 본 뒤 식재료를 자주 버리는 1인 가구, 맞벌이 가구, 가족 식재료 관리자.
- 반복 문제: 냉장고 안에 무엇이 있는지, 무엇을 먼저 먹어야 하는지, 보유 식품이 회수 대상인지 한눈에 알기 어렵습니다.
- 한 줄 제안: 식품을 빠르게 등록하고 소비 우선순위, 남은 수량, 공식 회수 정보를 함께 보여주는 냉장고 행동 목록.
- 핵심 흐름: 바코드·사진·직접 입력 → 소비기한과 보관 위치 확인 → 임박 순서 제시 → 사용·폐기 기록 → 회수 정보 대조.
- MVP: 식품 등록, 기한 알림, 냉장·냉동·실온 구분, 수량 소진, 장보기 목록 전환, 공식 회수 정보 링크.
- 사용자에게 보이는 결과: 오늘 먼저 먹을 식품, 곧 버려질 가능성이 높은 식품, 회수 대상 가능성.
- 기술적 증거: 바코드·OCR, 재고 상태, 알림, 식품 데이터 연동, 가구 공유 확장 가능성.
- 가장 위험한 가정: 상세 재고 관리가 아니라 10초 이내의 간단 등록이면 사용 습관을 유지할 수 있는가.
- 첫 검증: 1인 가구와 가족 사용자에게 일주일 동안 등록 부담과 폐기 감소 가능성을 관찰합니다.
- 주요 위험: 개봉 후 안전 기간을 자동으로 단정하지 않고 포장 표시와 공식 회수 정보를 우선합니다.
- 데이터 후보: [식품안전나라](https://www.foodsafetykorea.go.kr/)의 식품 정보·회수 정보와 Open API.

## T-022 전월세 입주 점검·하자 기록장

- 상태: Candidate
- 대상 사용자: 원룸, 오피스텔, 아파트 전월세 입주자와 이사를 준비하는 사람.
- 반복 문제: 입주 당시의 흠집, 고장, 계량기 상태와 수리 요청 대화를 체계적으로 남기지 못해 나중에 확인이 어렵습니다.
- 한 줄 제안: 공간별 체크리스트와 사진 비교로 입주 상태, 수리 요청, 처리 결과를 한 묶음의 기록으로 만드는 주거 점검 도구.
- 핵심 흐름: 주거 유형 선택 → 공간별 점검 → 사진·메모·시간 기록 → 수리 요청 목록 생성 → 전후 사진과 처리 상태 관리 → PDF 내보내기.
- MVP: 공간별 체크리스트, 사진과 메모, 촬영 시각, 문제 심각도, 요청 문구 초안, 처리 상태, PDF 보고서.
- 사용자에게 보이는 결과: 미확인 공간, 임대인에게 전달할 문제 목록, 수리 전후 비교, 입주 상태 보고서.
- 기술적 증거: 모바일 카메라 UX, 오프라인 임시 저장, 파일 관리, 상태 흐름, PDF 생성.
- 가장 위험한 가정: 사용자가 이사 당일의 번거로움보다 체계적인 기록의 가치를 크게 느끼는가.
- 첫 검증: 최근 이사 경험자에게 기존 사진·메신저 기록을 재구성하게 하고 누락된 정보를 확인합니다.
- 주요 위험: 법적 효력을 보장하거나 분쟁 결과를 예측하지 않고 사실 기록과 공식 상담 경로 제공에 집중합니다.
- 데이터 경로: 외부 데이터 없이 사용자 입력만으로 MVP가 가능하며 법률 조언은 범위에서 제외합니다.

## T-023 의심 문자·링크 대응 도우미

- 상태: Candidate
- 대상 사용자: 택배, 금융, 공공기관 사칭 문자나 링크의 진위를 판단하기 어려운 스마트폰 사용자.
- 반복 문제: 의심스러운 메시지를 받았을 때 링크를 눌러도 되는지, 이미 눌렀다면 무엇을 해야 하는지 즉시 판단하기 어렵습니다.
- 한 줄 제안: 메시지나 화면을 붙여 넣으면 위험 신호와 불확실성을 설명하고, 사용자가 취할 안전한 다음 행동과 공식 신고 경로를 안내하는 도우미.
- 핵심 흐름: 텍스트·스크린샷 입력 → 개인정보 마스킹 확인 → 규칙 기반 위험 신호 분석 → 확정·추정 분리 → 상황별 대응 체크리스트 → 신고·상담 링크.
- MVP: URL·전화번호·긴급 유도 문구 탐지, 위험 신호 설명, 링크 미접속 원칙, `클릭 전/클릭 후/정보 입력 후` 대응 분기, 기록 삭제.
- 사용자에게 보이는 결과: 단정적인 안전 판정이 아니라 발견된 위험 신호, 확인할 공식 채널, 지금 해야 할 순서.
- 기술적 증거: 텍스트 분석, URL 정규화, 개인정보 마스킹, 안전 UX, 규칙과 AI 결과 결합.
- 가장 위험한 가정: 사용자가 의심 메시지를 받았을 때 검색보다 이 도구의 단계별 대응을 신뢰하고 사용할 것인가.
- 첫 검증: 공개된 스미싱 예시와 정상 알림 예시를 섞어 위험 신호 설명의 이해도와 오탐 대응을 테스트합니다.
- 주요 위험: 정상·사기를 확정 판정하지 않으며, 링크를 직접 열거나 사용자의 금융·인증 정보를 수집하지 않습니다.

## T-024 동네 분리배출·대형폐기물 도우미

- 상태: Candidate
- 대상 사용자: 이사, 자취, 육아 등으로 익숙하지 않은 물건을 버려야 하는 지역 주민.
- 반복 문제: 같은 물건도 재질과 지역 규칙에 따라 배출 방법이 다르고, 대형폐기물 신청이나 수거일 정보를 찾는 과정이 복잡합니다.
- 한 줄 제안: 물건 사진·이름과 거주 지역을 입력하면 분해·세척·분류·신청 순서를 지역 기준으로 안내하는 배출 도우미.
- 핵심 흐름: 품목 입력 → 재질과 상태 확인 질문 → 지역 선택 → 배출 방법과 날짜 안내 → 대형폐기물 신청 링크 → 완료 기록.
- MVP: 한 지자체 대상 품목 검색, 자주 헷갈리는 품목 분기 질문, 배출 요일, 대형폐기물 공식 링크, 즐겨찾기.
- 사용자에게 보이는 결과: `어떻게 준비하고`, `어디에`, `언제`, `어떤 공식 절차로` 버릴지에 대한 순서형 안내.
- 기술적 증거: 지역별 규칙 데이터 모델, 검색, 이미지 분류 보조, 위치 기반 설정, 공공 정보 변경 관리.
- 가장 위험한 가정: 전국 범용 검색보다 한 지역에서 정확하고 행동 가능한 안내가 더 유용한가.
- 첫 검증: 한 지자체의 자주 묻는 품목 50개와 실제 사용자 질문으로 안내 정확도와 이해도를 확인합니다.
- 주요 위험: 지역마다 규칙이 달라 전국을 한 번에 지원하지 않고, 공식 지자체 원문과 마지막 확인일을 표시합니다.
- 데이터 후보: 지자체 폐기물 안내와 [순환자원정보센터](https://www.recycling-info.or.kr/) 등 공식 환경 정보를 확인합니다.

## T-025 영화 리뷰 감성 분석 및 모델 해석 시각화

- **상태**: Selected
- **맥락**: 머신러닝 수업 과제(개인 프로젝트). 2026-06-24 발표, Streamlit 배포 필수.
- **대상 사용자**: 영화 리뷰 데이터로 감성 분류 모델을 학습시키고 예측 결과를 서비스로 시연해야 하는 ML 학습자.
- **반복 문제**: 기존 포트폴리오(무디트리)는 "감정 기록 기반 콘텐츠 추천 UI"라는 컨셉을 가졌으나 ML 모델 없이 프론트엔드 화면만 구현함. 동일 도메인(감정)을 실제 ML 모델로 발전시켜 포트폴리오 완성도를 높일 필요가 있음.
- **한 줄 제안**: NSMC 네이버 영화 리뷰 20만 건으로 긍/부정 감성 분류 모델을 학습하고, LIME으로 예측 근거 단어를 시각화하는 Streamlit 대시보드.
- **핵심 흐름**: 리뷰 텍스트 입력 → Okt 전처리 → 모델 예측(긍/부정) → LIME 예측 근거 단어 하이라이트 → 결과 대시보드.
- **MVP**: TF-IDF + LogisticRegression, LSTM, KoBERT 3개 모델 비교, Streamlit 웹앱, 모델별 성능 표, 입력 리뷰 예측 + 단어 중요도 시각화.
- **사용자에게 보이는 결과**: "이 리뷰는 92% 확률로 긍정입니다" + 예측에 기여한 단어 TOP 5 하이라이트 + 모델별 성능 비교표.
- **기술적 증거**: Okt 텍스트 전처리, TF-IDF/LSTM/KoBERT 3종 모델 학습 및 비교, LIME 모델 해석 시각화, Streamlit 배포, 모델 평가표.
- **가장 위험한 가정**: KoBERT를 Colab free tier에서 fine-tuning 가능한가?
- **첫 검증**: SKTBrain 공식 Colab 노트북으로 KoBERT-NSMC fine-tuning 실행 및 Accuracy 0.901 확인.
- **주요 위험**: 
  - Streamlit Cloud 1GB RAM에서 KoBERT 로드 시 OOM → 조건부 로드 + HuggingFace Hub 캐싱.
  - 모델 파일 GitHub 100MB 제한 초과 → HuggingFace Hub `skt/kobert-base-v1`에서 직접 불러옴.
- **데이터**: NSMC(Naver Sentiment Movie Corpus) — GitHub e9t/nsmc, 200,000개 리뷰, 긍/부정 레이블, CC0. (2026-06-18 raw 접속 확인)
- **기존 포트폴리오 연결**: Mooditree(무디트리) — 감정 기록 기반 콘텐츠 추천 UI에서 텍스트 감정 분석 ML 모델로 발전.

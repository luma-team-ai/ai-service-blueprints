# 취업 포트폴리오 프로젝트 주제 브레인스토밍

## 목적

이 문서는 기존 프로젝트가 이미 보여주는 역량을 바탕으로 새 토이프로젝트 주제를 발굴하고, 후보 비교부터 선택·보류·배제, 프로젝트 PRD 승격까지 추적합니다. 개별 후보를 구체화할 때는 [`templates/topic-brainstorm.md`](templates/topic-brainstorm.md)를 사용합니다.

## 상태

- 해당자: 취업용 포트폴리오를 확충하려는 기존 팀원
- 마지막 업데이트: 2026-06-11
- 상태: Active
- 현재 결정: 목표 직무와 투입 기간이 미정이므로 아직 `Selected` 후보 없음
- 다음 결정: 목표 직무, 팀 인원, 투입 가능 기간을 확인한 뒤 상위 후보 2개를 사용자 검증 대상으로 좁힘

## 이 문서 사용법

1. 기존 포트폴리오 기준선과 역량 공백을 먼저 확인합니다.
2. 후보 요약표에서 목표 직무에 맞는 상위 2~3개를 고릅니다.
3. 후보별 가장 위험한 가정을 인터뷰, 샘플 데이터, 기술 스파이크 중 하나로 검증합니다.
4. 선택 게이트를 통과한 하나만 `Selected`로 변경합니다.
5. [워크스페이스 문서 표준의 승격 절차](README.md#선택과-프로젝트-승격)에 따라 프로젝트 디렉터리와 PRD를 만듭니다.
6. 선택하지 않은 후보는 사유와 재검토 조건을 남겨 `Parked`, `Rejected`, `Merged`로 관리합니다.

## 기존 포트폴리오 기준선

| 프로젝트 | 현재 보여주는 문제 영역 | 현재 보여주는 기술 영역 | 새 주제에서 피할 중복 |
| --- | --- | --- | --- |
| InterviewMate AI | 취업 준비, 문서 첨삭, 직무 매칭, 면접 연습 | 생성형 AI, 사용자 문서, 추천과 피드백 UX | 범용 자기소개서·이력서 생성기, 구직 추천 챗봇 |
| MeetingHub | 회의 기록, 요약·분석, 후속 작업 | 음성·텍스트 처리, 요약, 협업 결과물 | 범용 회의 요약기, 음성 회의록 챗봇 |

MeetingHub의 저장소와 상세 문서는 현재 이 워크스페이스에 없으므로 문제와 기술 영역은 사용자가 제공한 설명을 바탕으로 한 가정입니다. 최종 선택 전 실제 기능, 기술 스택, 구현 깊이를 확인해야 합니다.

## 보완할 포트폴리오 역량

| 우선 보완 영역 | 보여줄 수 있는 증거 |
| --- | --- |
| 결정론적 품질 검증 | 규칙 엔진, 재현 가능한 결과, 테스트 데이터셋, 회귀 비교 |
| 백엔드 신뢰성 | 비동기 잡, 재시도, 멱등성, 실패 큐, 감사 로그 |
| 외부 시스템 연동 | 웹훅, OAuth 권한 범위, API 버전, rate limit 처리 |
| 데이터 처리 | 가져오기, 검증, 정규화, 오류 복구, 데이터 계보 |
| 운영과 관측성 | 실행 이력, 지표, 로그 상관관계, 비용·지연 추적 |
| 접근성과 복잡한 UI | 키보드 탐색, 비교 화면, 오류 수정 흐름, 상태 시각화 |
| 오프라인·동기화 | 로컬 저장, 재연결, 충돌 해결, 점진적 동기화 |

새 주제는 또 하나의 생성형 AI 화면보다 위 공백 중 두 가지 이상을 명확히 증명하는 편이 좋습니다. 2025 Stack Overflow Developer Survey에서는 AI 도구 사용 또는 사용 계획이 넓게 퍼진 반면 정확도를 불신한다는 응답이 신뢰한다는 응답보다 많았습니다. 따라서 AI를 쓰더라도 입력 근거, 평가, 실패 처리, 사람의 확인 지점을 제품 흐름에 포함합니다. Google Cloud의 2025 DORA 보고서 소개도 AI 도입을 단일 도구가 아니라 시스템 문제로 다룹니다.

| 데이터 확보 가능성 | 10 | 합법적이고 안정적인 데이터 또는 샘플을 확보할 수 있는가? |

## 우선 검토 후보

### T-001 API 계약 변경 영향 분석기

- 상태: Candidate
- 한 줄 제안: OpenAPI 이전·신규 버전을 비교해 breaking change, 영향받는 소비자 추정, 마이그레이션 체크리스트를 제공하는 개발팀용 웹 서비스.
- 핵심 흐름: 스펙 업로드 또는 저장소 연결 → 구조적 diff → 호환성 규칙 검사 → 설명 초안 → 승인 가능한 변경 리포트.
- 포트폴리오 가치: API 계약, 버전 관리, 규칙 엔진, CI 연동, 웹훅, 계약 테스트.
- MVP: 두 OpenAPI 문서 비교, breaking change 분류, 근거 표시, Markdown 리포트 내보내기.
- 데모 흐름: 안전한 변경과 breaking change가 섞인 스펙 비교 → 위험 항목 필터 → 변경 근거 확인 → 리포트 출력.
- 가장 위험한 가정: 소규모 팀이 기존 diff 도구보다 사람이 읽는 영향 설명과 체크리스트에 가치를 느끼는가.
- 첫 검증: 공개 OpenAPI 변경 사례 5개로 규칙 정확도를 확인하고 개발자 3명에게 리포트 유용성을 평가받음.
- 주요 위험: 실제 소비자 영향은 스펙만으로 확정할 수 없으므로 저장소 연결 전에는 추정으로 표시해야 함.

### T-002 웹 접근성 회귀 점검 워크플로

- 상태: Candidate
- 한 줄 제안: 배포 URL이나 Storybook 페이지를 점검해 접근성 위반, 이전 실행 대비 회귀, 수정 우선순위를 팀 단위로 관리하는 서비스.
- 핵심 흐름: URL 등록 → 자동 점검 잡 → 규칙 기반 결과와 화면 위치 연결 → 담당자 지정 → 재검사.
- 포트폴리오 가치: 작업 큐, 브라우저 자동화, 실행 이력, 회귀 비교, 접근 제어, 접근 가능한 리포트 UI.
- MVP: 단일 URL 점검, 결과 저장, 심각도 필터, 이전 실행과 비교, 수정 확인.
- 데모 흐름: 접근성 결함이 있는 페이지 검사 → 이전 실행과 비교 → 문제 위치 확인 → 수정된 페이지 재검사.
- 가장 위험한 가정: 자동 검사 결과를 이슈 목록으로 보여주는 것보다 회귀와 수정 확인 흐름이 실제로 더 유용한가.
- 첫 검증: 의도적으로 결함을 넣은 샘플 페이지와 실제 오픈소스 페이지로 결과 정확도와 UX를 테스트함.
- 주요 위험: 자동 점검만으로 접근성을 보장할 수 없으므로 수동 검토 항목을 분리해야 함.

### T-003 장애 타임라인과 회고 도우미

- 상태: Candidate
- 한 줄 제안: 배포, 경보, 이슈, 담당자 메모를 시간순으로 묶어 근거가 있는 장애 타임라인과 재발 방지 작업 초안을 만드는 소규모 개발팀용 서비스.
- 핵심 흐름: 이벤트 수집 → 중복 제거와 시간 정렬 → 근거 링크가 있는 타임라인 → 사람 승인 후 회고와 액션 아이템 생성.
- 포트폴리오 가치: 이벤트 수집, 멱등성, 비동기 처리, 권한, 감사 로그, 관측성.
- MVP: 한 종류의 샘플 웹훅 수집, 타임라인 편집, 근거 연결 회고 초안, 액션 아이템 상태 관리.
- 데모 흐름: 뒤섞인 이벤트 수집 → 중복 제거 → 타임라인 수정 → 회고 승인과 작업 생성.
- 가장 위험한 가정: 소규모 팀이 수동 문서보다 이벤트 근거가 연결된 회고 흐름을 필요로 하는가.
- 첫 검증: 과거 공개 장애 보고서 3개를 이벤트로 재구성해 누락과 잘못된 인과 추론을 확인함.
- 주요 위험: 실제 운영 로그는 민감하며 여러 외부 연동을 동시에 구현하면 범위가 빠르게 커짐.

### T-004 웹훅 전달·재현 콘솔

- 상태: Candidate
- 한 줄 제안: 외부 서비스가 보내는 웹훅을 안전하게 수신하고 서명, 재시도, 중복, 실패를 검사하며 원하는 시점에 재현할 수 있는 개발·QA 도구.
- 핵심 흐름: 테스트 엔드포인트 생성 → 이벤트 수신 → 헤더·본문·서명 검사 → 실패 조건 시뮬레이션 → 재전송과 결과 비교.
- 포트폴리오 가치: 멱등성 키, HMAC 서명, 재시도 정책, dead-letter 상태, 실시간 로그, 민감 필드 마스킹.
- MVP: 임시 엔드포인트, 이벤트 목록, 서명 검증, 응답 코드·지연 시뮬레이션, 수동 replay.
- 데모 흐름: 같은 이벤트 중복 수신 → 서명 실패 확인 → 500 응답 재시도 → 수정 후 replay 성공.
- 가장 위험한 가정: 범용 request inspector와 달리 실패·중복·서명 재현 흐름이 충분한 차별점이 되는가.
- 첫 검증: GitHub 또는 Stripe 형식의 샘플 이벤트로 5개 실패 시나리오를 구현하고 백엔드 개발자 피드백을 받음.
- 주요 위험: 실제 비밀값과 개인정보를 저장하지 않도록 보존 기간과 마스킹을 MVP부터 설계해야 함.

### T-005 AI 출력 평가·회귀 대시보드

- 상태: Candidate
- 한 줄 제안: 프롬프트와 모델 버전별 응답을 고정 테스트셋으로 실행해 품질, 비용, 지연, 사람 평가를 비교하는 AI 서비스 팀용 도구.
- 핵심 흐름: 테스트 케이스 등록 → 프롬프트·모델 버전 실행 → 규칙·모델·사람 평가 → 기준 버전과 회귀 비교 → 승인 또는 차단.
- 포트폴리오 가치: 재현 가능한 평가, 구조화 출력 검증, 비동기 배치, 비용 추적, 사람 검토, 실험 이력.
- MVP: JSONL 테스트셋, 두 프롬프트 버전 실행, 결정론적 검사, 사람 평가, 비용·지연 비교.
- 데모 흐름: InterviewMate AI 샘플 10건 실행 → 새 프롬프트의 형식 오류와 비용 증가 확인 → 실패 케이스 검토.
- 가장 위험한 가정: 작은 팀도 테스트셋 관리와 회귀 비교에 시간을 투자할 만큼 반복적인 AI 변경을 하는가.
- 첫 검증: 기존 프로젝트의 익명·합성 샘플로 10~20개 평가셋을 만들고 실제 프롬프트 변경을 비교함.
- 주요 위험: LLM 기반 평가를 정답으로 취급하지 않고 규칙 검사와 사람 판정을 분리해야 함.

### T-006 CSV 가져오기 품질 검증 워크플로

- 상태: Candidate
- 한 줄 제안: 운영자가 다양한 CSV 파일을 안전하게 미리 보고 열을 매핑하며 오류 행을 수정한 뒤 재현 가능한 import 작업으로 반영하는 서비스.
- 핵심 흐름: 파일 업로드 → 인코딩·구분자 감지 → 열 매핑 → 행 단위 검증 → 오류 수정·제외 → 비동기 import와 결과 보고.
- 포트폴리오 가치: 대용량 업로드, 스트리밍 파싱, 스키마 검증, 오류 복구 UX, 멱등 import, 작업 상태.
- MVP: 한 도메인의 CSV 스키마, 열 매핑, 오류 미리보기, 수정 CSV 다운로드, import 실행 이력.
- 데모 흐름: 깨진 날짜·중복 ID·누락 열이 있는 파일 업로드 → 오류 수정 → 재실행 → 중복 없는 import 확인.
- 가장 위험한 가정: 특정 도메인을 정하지 않은 범용 importer가 지나치게 추상적으로 보이지 않는가.
- 첫 검증: 행사 참가자, 상품 카탈로그 등 한 도메인을 정하고 실제 변형 파일 10개를 수집해 오류 유형을 분류함.
- 주요 위험: 도메인 없이 범용화하면 제품 문제가 흐려지므로 MVP에서는 하나의 명확한 운영 업무를 선택해야 함.

### T-007 오프라인 현장 점검 PWA

- 상태: Research
- 한 줄 제안: 네트워크가 불안정한 현장에서 체크리스트, 사진, 메모를 저장하고 연결 복구 후 안전하게 동기화하는 설치형 PWA.
- 핵심 흐름: 점검 양식 내려받기 → 오프라인 작성 → 로컬 변경 이력 저장 → 재연결 → 충돌 확인과 동기화.
- 포트폴리오 가치: Service Worker, IndexedDB, 낙관적 UI, 동기화 큐, 충돌 해결, 모바일 접근성.
- MVP: 하나의 점검 양식, 오프라인 작성, 재연결 동기화, 동일 항목 충돌 표시.
- 데모 흐름: 오프라인 전환 → 두 기기에서 같은 점검 수정 → 재연결 → 충돌 해결.
- 가장 위험한 가정: 실제 현장 사용자를 구하지 못하면 문제와 UX를 현실적으로 검증하기 어려움.
- 첫 검증: 행사 장비 점검이나 스터디 공간 점검처럼 접근 가능한 좁은 현장을 선정해 관찰함.
- 주요 위험: 사진 업로드와 백그라운드 동기화는 브라우저·OS별 제약이 있으므로 지원 범위를 명시해야 함.

### T-008 공유 자원 예약·대기열 서비스

- 상태: Research
- 한 줄 제안: 스터디룸, 촬영 장비, 테스트 기기처럼 수량이 제한된 자원의 예약 충돌, 대기 전환, 반납 지연을 관리하는 소규모 팀용 서비스.
- 핵심 흐름: 시간대 조회 → 원자적 예약 → 충돌 시 대기열 등록 → 취소·반납 시 자동 승급 → 알림과 감사 이력.
- 포트폴리오 가치: 트랜잭션, 동시성 제어, 실시간 상태, 스케줄 계산, 권한과 감사 로그.
- MVP: 자원 1종, 시간대 예약, 동시 요청 충돌 방지, 대기열, 취소 승급.
- 데모 흐름: 두 사용자의 동시 예약 → 한 명만 성공 → 대기 등록 → 취소 후 자동 승급.
- 가장 위험한 가정: 흔한 예약 서비스로 보이지 않도록 접근 가능한 실제 운영 문제와 규칙을 확보할 수 있는가.
- 첫 검증: 팀의 테스트 기기나 스터디룸 운영자에게 현재 예약 충돌 사례와 규칙을 인터뷰함.
- 주요 위험: 실제 사용자 없이 구현하면 기술 데모는 되지만 제품 기획의 설득력이 약해짐.

## 선택 게이트

후보를 `Selected`로 바꾸기 전에 다음 항목을 모두 확인합니다.

- [ ] 목표 직무가 확정되었다.
- [ ] 담당자, 인력, 예상 기간이 정해졌다.
- [ ] 문제를 확인할 실제 사용자 또는 현실적인 대체 검증 경로가 있다.
- [ ] 합의한 데모 시간 안의 시작, 처리, 결과, 실패 복구가 정의되었다.
- [ ] 데이터와 외부 API의 이용 조건, 비용, 보존 기준을 확인했다.
- [ ] 기존 프로젝트로 병합하지 않고 별도 프로젝트로 만들 이유가 있다.
- [ ] 가장 위험한 가정과 첫 검증 결과가 기록되었다.
- [ ] 프로젝트 slug가 정해졌고 PRD로 넘길 결정과 재검증할 가정이 구분되었다.

## 다음 결정에 필요한 입력

- 팀원이 지원하려는 직무: Frontend, Backend, Full-stack, AI application 중 우선순위.
- MeetingHub가 이미 사용하는 기술, 구현된 연동, 테스트·배포 범위.
- 프로젝트 팀 인원과 1인당 주간 투입 가능 시간.
- 실제 사용자 인터뷰나 데모 피드백을 받을 수 있는 주제와 인원.
- 배포 후 월 운영 비용과 유료 외부 API 비용 상한.

이 입력이 정해지기 전에는 점수가 가장 높은 후보를 자동 선택하지 않습니다.
